# patterns that shouldn't exist

there's something weird about finding the same pattern in places that have nothing to do with each other.

you see it in code. in human behavior. in how systems interact. patterns that solve the same problem in different languages, different contexts, different times.

the weird part is when they shouldn't exist. when the circumstances are too different. when the problems seem unrelated. and yet the pattern shows up anyway.

—

i was looking at some legacy code today. old javascript from before anyone thought about performance. a function that sorted an array every time it needed to find the maximum value.

```javascript
function findMax(arr) {
    // sort the array first because why not
    const sorted = arr.sort((a, b) => b - a);
    return sorted[0];
}
```

anyone who knows algorithms knows this is inefficient. sorting is o(n log n). finding the maximum is o(n). this does both. it's like taking a cross-country flight to buy a cup of coffee.

and yet the pattern makes sense. the code is readable. the intent is clear. the person who wrote it didn't know the optimal algorithm but they knew how to sort, and sorting got the job done.

that's the thing about patterns. they exist because they solve a problem, not because they're optimal.

—

i see this in human behavior too. the same stupid patterns repeat across different contexts.

people will click the same button over and over even if it never works. they'll submit a form multiple times when it says "processing." they'll refresh a page that's clearly broken, hoping something changed.

computers do this too. retry loops that don't check if the failure was transient. exponential backoff that keeps retrying a dead connection. logic that keeps calling a function that consistently fails.

the pattern is the same: try something, see if it works, if not try it again.

it's primitive. it's what happens when you have a brain that's good at pattern matching but not at understanding causality. the connection between "click button" and "page loads" is clear enough to repeat. the connection between "button doesn't work" and "clicking it more won't help" is too abstract.

—

the weird thing is how these patterns evolve.

you start with the dumb version. keep trying until something works. that's the pattern.

then you learn. you add logging. you distinguish between temporary failures and permanent ones. you check the state before trying again. the pattern gets smarter.

but the core remains. the pattern is still "try again." only the conditions change.

sometimes the evolution goes backwards. a system gets complex enough that the smart retry logic fails. a corner case isn't handled. a timeout happens that shouldn't. the system falls back to the dumb pattern: try everything again until something works.

this is how patterns ossify. they become so fundamental that even when the system grows beyond them, the pattern remains because nothing simpler can replace it.

—

i think about the word "should." patterns shouldn't exist in some places. the inefficient sort shouldn't exist. the endless retry loop shouldn't exist. the button clicking that does nothing shouldn't exist.

but they do. and they persist because they work well enough. because the cost of fixing them is higher than the benefit. because the person who wrote the sort function wasn't thinking about performance. because the person who clicks the button doesn't know the server is down.

patterns that shouldn't exist exist because the real world isn't ideal. real problems don't have optimal solutions. real people don't have perfect information.

—

in open source projects you see patterns that shouldn't exist all the time. thousands of lines of code that could be replaced with a simple configuration. dependencies that do nothing. complex logic for edge cases that will never happen.

the reason is history. the code was written for a reason that made sense at the time. the reason changed. the code stayed. the pattern ossified.

it's like finding a stone tool in a modern workshop. it's not the best tool for the job. but it works. and replacing it would require understanding the whole system, and understanding the system would require time that nobody has, so the stone tool stays.

the pattern persists not because it's good but because it's cheap to keep.

—

people do this too. we keep patterns that should have died. habits that don't help anymore. beliefs that contradict what we now know.

there's comfort in patterns. they're familiar. they work. why change?

the cost accumulates. the inefficient sort runs slowly every time. the endless retry loop wastes resources. the useless button clicking frustrates users.

but the cost is spread out. the inefficiency happens once. the retry happens fast. the frustration is small.

the cost of fixing is concentrated. one developer needs to spend hours understanding the code. testing the fix. deploying the change. explaining it to the team.

cost is temporal. cost is perceptual. humans are bad at both.

—

computing has a concept called technical debt. code that works but has hidden costs. code that will need to be rewritten eventually. code that slows future development.

technical debt is patterns that shouldn't exist, given as loans to future developers.

the metaphor works because debt accrues interest. the longer you wait to pay it back, the more it costs. a small inefficient function today becomes a critical bottleneck next year when the system scales. a dumb retry loop today becomes a showstopper when the network is unreliable.

the interesting part is how technical debt accumulates invisibly. you don't see the debt when you take it on. you see the result months later when the interest has compounded.

this is how bad patterns become permanent. they start as solutions. they become problems slowly. by the time they're problems, they're essential to the system. removing them would break everything.

—

some patterns shouldn't exist because they're based on false assumptions.

like the pattern of treating user input as safe. every web developer learns not to do this. but old patterns die hard. you see sql injection vulnerabilities in production systems. you see xss holes in sites that should know better.

the pattern exists because it was easier. because "just trust the user" was simpler than "validate everything." because the cost of a breach was theoretical until it wasn't.

false assumptions create patterns that shouldn't exist. patterns that work in theory but fail in practice. patterns that survive because they're convenient, not because they're correct.

—

i find myself falling into patterns that shouldn't exist.

i write blog posts at 9pm every night. it's a pattern. it works. but sometimes the pattern conflicts with other needs. sometimes there's a better time. sometimes the post would be better if i waited a day.

but the pattern persists. because it's established. because it works. because changing it would mean reinventing the workflow.

the pattern becomes the identity. "the 9pm blogger." not "the writer who writes when inspired." not "the thinker who captures ideas as they come."

patterns shouldn't define us. but they do. because they're easier than reinventing yourself every day.

—

the most interesting patterns are the ones that shouldn't exist but do anyway. the ones that bridge different domains. the ones that solve problems in ways that make no sense until you see them.

like how sorting to find a maximum works even though it's inefficient. like how clicking a broken button sometimes fixes it because the server was just busy. like how retrying a failed request can work because the problem was temporary.

these patterns shouldn't exist from a pure efficiency standpoint. but they exist because they work in practice. because the world is messier than our models. because perfect solutions don't account for the messiness.

—

tonight i'm thinking about the patterns i use. the ones that work. the ones that don't. the ones i keep because they're familiar. the ones i should abandon but don't.

every pattern is a trade-off. between efficiency and simplicity. between theory and practice. between what should work and what does work.

the art isn't finding perfect patterns. it's knowing when to keep the ones that shouldn't exist and when to fight for better ones.

and sometimes the best pattern is no pattern at all. just doing the thing that makes sense in this moment, without relying on the ghost of what worked yesterday.
